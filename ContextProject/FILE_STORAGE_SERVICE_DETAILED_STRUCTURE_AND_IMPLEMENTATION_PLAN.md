# AutoShop FileStorageService: подробный план-описание структуры проекта и примерного хода реализации

Дата подготовки документа: 2026-05-13  
Проект: `autoshop-files`  
Тип проекта: отдельный backend-микросервис для хранения файлов  
Основной стек: `Java 17`, `Spring Boot 3`, `Spring Web`, `Spring Data JPA`, `Liquibase`, `PostgreSQL`, `MinIO`, `Docker Compose`, `JUnit 5`, `Mockito`

---

## 1. Назначение проекта

`AutoShop FileStorageService` — это выделенный микросервис, который отвечает за хранение бинарных файлов и их метаданных в рамках общей микросервисной системы AutoShop.

Главная идея проекта:

- сами **байты файлов** хранятся в `MinIO`;
- **метаданные файлов** хранятся в `PostgreSQL`;
- остальные сервисы системы не должны знать детали `bucket`, `object key`, `presigned URL`, правила S3-хранилища и внутреннюю логику размещения файлов;
- внешние сервисы работают только через HTTP API `FileStorageService` и сохраняют у себя лишь `fileId` либо связь `ownerType + ownerId`.

Это архитектурное разделение прямо зафиксировано в `README.md` и подтверждается кодовой структурой проекта.

---

## 2. Место проекта в общей системе AutoShop

Данный сервис не является монолитной частью `autoshop-core`, а проектируется как самостоятельный инфраструктурно-доменный модуль.

Его роль в общей системе:

- принимать загрузку документов, фотографий, вложений и аватаров;
- валидировать тип файла, размер, расширение и допустимость имени;
- формировать безопасный внутренний путь хранения;
- сохранять файл в объектное хранилище `MinIO`;
- сохранять связанную запись о файле в `PostgreSQL`;
- отдавать метаданные по `fileId`;
- отдавать содержимое файла через прямой download endpoint;
- генерировать временные `presigned download URL`;
- предоставлять список файлов по владельцу (`ownerType`, `ownerId`);
- выполнять soft delete без потери истории метаданных.

С точки зрения дипломного описания этот сервис выступает как **изолированный file boundary** между бизнес-модулями и физическим файловым хранилищем.

## 3. Общая архитектурная идея

Проект построен по слоистой backend-архитектуре с чётким разделением ответственности.

### 3.1. Верхнеуровневая схема

```text
Клиент / другой микросервис
          |
          v
     REST Controller
          |
          v
       FileService
      /    |     \
     /     |      \
Validation  DB    Object Storage
   layer   layer      layer
     |       |          |
     v       v          v
  rules   PostgreSQL   MinIO
```

### 3.2. Главные архитектурные принципы

1. **Разделение байтов и метаданных**  
   Бинарные данные не хранятся в БД. БД содержит только описание файла, его владельца, путь, контрольную сумму, статус и служебные поля.

2. **Скрытие инфраструктурных деталей**  
   Внешний клиент не знает названия bucket, object key и внутренний путь размещения файла.

3. **Категория файла как публичный контракт**  
   Клиент работает с `category`, а не с физическим bucket. Сопоставление категории и bucket инкапсулировано в enum `FileCategory`.

4. **Жизненный цикл файла через статусы**  
   Проект моделирует жизненный цикл загрузки: файл может быть доступным, недоступным после ошибки, либо мягко удалённым.

5. **Готовность к будущей интеграции с Auth/Core**  
   В проекте уже есть граница `FileAccessPolicy`, хотя текущая реализация допускает все операции.

6. **Минимальная связность с другими сервисами**  
   Сервис знает только `ownerType` и `ownerId`, не подтягивая доменные сущности из Core напрямую.

## 5.2. Пакет `file` — внешний API и основная бизнес-логика

Этот пакет является центральным слоем проекта.

### `FileController`

Назначение:

- публикует REST API по пути `/api/files`;
- принимает multipart upload;
- возвращает метаданные;
- выполняет list by owner;
- отдает download;
- генерирует presigned URL;
- выполняет delete.

Фактические endpoint'ы:

- `POST /api/files`
- `GET /api/files/{fileId}`
- `GET /api/files`
- `GET /api/files/{fileId}/download`
- `POST /api/files/{fileId}/presigned-download-url`
- `DELETE /api/files/{fileId}`

Важные особенности контроллера:

- использует enum `FileCategory` как входной параметр категории;
- использует enum `OwnerType` как тип владельца файла;
- поддерживает пагинацию списка файлов;
- для download выставляет `Content-Disposition` и `Content-Type`;
- вход на presigned endpoint валидируется через DTO `PresignedUrlRequest`.

### `FileService`

Это главный orchestration-layer проекта.

Именно здесь сосредоточена основная логика:

- валидация `ownerId`;
- вызов `FileAccessPolicy`;
- валидация файла через `FileUploadValidator`;
- генерация `UUID` файла;
- вычисление `SHA-256` контрольной суммы;
- определение bucket через `FileCategory`;
- генерация безопасного `objectKey` через `ObjectKeyFactory`;
- создание и сохранение `FileMetadata` в БД;
- загрузка байтов в `MinIO` через `ObjectStorageService`;
- перевод статуса файла в `AVAILABLE` или `UPLOAD_FAILED`;
- выдача метаданных и выдача download stream;
- список файлов владельца;
- генерация presigned URL;
- soft delete;
- cleanup объекта в `MinIO` в случае ошибки.

Этот класс отражает важную для диплома идею: **все внешние запросы проходят через единый сервисный сценарий**, где согласованно меняются и БД, и объектное хранилище.

### `FileStatus`

Enum жизненного цикла файла.

Фактически в проекте используются статусы:

- `PENDING` — метаданные созданы, физическая загрузка ещё не подтверждена;
- `AVAILABLE` — файл успешно загружен и доступен;
- `UPLOAD_FAILED` — запись создана, но загрузка в объектное хранилище завершилась ошибкой;
- `DELETED` — файл мягко удалён.

Этот подход позволяет корректно моделировать отказоустойчивость загрузки.

### `OwnerType`

Enum, определяющий владельца файла на уровне доменной связи.

Поддерживаемые типы владельца:

- `VEHICLE`
- `ORDER`
- `CUSTOMER`
- `CLIENT`
- `EMPLOYEE`
- `PART`
- `PURCHASE_ORDER`
- `SYSTEM`

Важно, что сервис не знает саму сущность владельца, а лишь сохраняет ссылку в абстрактном виде. Это снижает связанность между микросервисами.

### DTO-пакет `file/dto`

Содержит транспортные модели API:

- `FileMetadataResponse` — ответ по одному файлу;
- `OwnerFilesResponse` — ответ списка файлов с пагинацией;
- `PresignedUrlRequest` — запрос на генерацию presigned URL;
- `PresignedUrlResponse` — ответ с URL, временем истечения и TTL.

DTO-слой отделяет HTTP-контракт от entity-модели БД.

## 5.3. Пакет `metadata` — модель и слой доступа к БД

### `FileMetadata`

Это центральная JPA entity проекта, представляющая метаданные файла.

Сущность хранит:

- `id` — UUID файла;
- `category` — логическая категория файла;
- `ownerType` и `ownerId` — связь с владельцем;
- `uploadedBy` — кто загрузил файл;
- `bucketName` — фактическое хранилище;
- `objectKey` — внутренний путь в MinIO;
- `originalFilename` — сохранённое имя файла после санации;
- `contentType` — MIME-тип;
- `sizeBytes` — размер файла;
- `checksumSha256` — контрольная сумма;
- `etag` — идентификатор/метка объекта со стороны S3-совместимого хранилища;
- `status` — состояние файла;
- `createdAt`, `updatedAt`, `deletedAt` — служебные timestamp-поля.

Также у сущности есть методы управления жизненным циклом:

- `markAvailable()`
- `markUploadFailed()`
- `markDeleted()`
- `isAvailable()`
- `isDeleted()`

Именно эта сущность показывает, что проект хранит не только ссылку на файл, но и полноценный аудит состояния.

### `FileMetadataRepository`

Spring Data JPA репозиторий.

Используется для:

- поиска файла по `id` и статусу;
- выдачи списка по `ownerType + ownerId`;
- фильтрации по категории;
- выборки с учётом или без учёта удалённых файлов;
- сортировки по времени создания.

Показательно, что проект использует **derived query methods**, а не ручной SQL, что ускоряет реализацию типовых поисковых сценариев.

### `FileMetadataMapper`

Слой преобразования entity -> DTO.

Назначение:

- не отдавать наружу JPA entity напрямую;
- формировать `FileMetadataResponse` из внутренней модели.

## 5.4. Пакет `storage` — адаптер к объектному хранилищу

### `ObjectStorageService`

Это абстракция над объектным хранилищем.

Методы интерфейса:

- `upload(...)`
- `download(...)`
- `remove(...)`
- `presignedGetUrl(...)`

Архитектурно это даёт следующие преимущества:

- возможность подмены `MinIO` на другой S3-совместимый storage;
- упрощение тестирования;
- отделение бизнес-логики от конкретной библиотеки `MinioClient`.

### `MinioObjectStorageService`

Конкретная реализация `ObjectStorageService` через `MinioClient`.

Реализует:

- `putObject` для загрузки;
- `getObject` для скачивания;
- `removeObject` для удаления;
- `getPresignedObjectUrl` для временной ссылки.

Все исключения инфраструктурного слоя оборачиваются в `StorageUnavailableException`, что делает поведение сервиса единообразным для верхних слоёв.

### `StoredObject`

Компактная транспортная модель для скачивания:

- `InputStream`
- `contentType`
- `size`

Она используется контроллером для формирования HTTP download response.

### `MinioHealthIndicator`

Компонент интеграции с `Spring Actuator`.

Назначение:

- проверить доступность `MinIO`;
- убедиться, что основной bucket существует;
- отразить состояние хранилища в `/actuator/health`.

Это показывает ориентацию проекта не только на функциональность, но и на эксплуатацию.

## 5.5. Пакет `validation` — валидация загрузки и подготовка object key

### `FileUploadValidator`

Основной валидатор входящих файлов.

Проверяет:

- что файл не пустой;
- что для категории существует правило;
- что размер файла укладывается в лимит;
- что расширение допустимо;
- что MIME-тип допустим;
- что имя файла прошло санацию.

Результат валидации возвращается как `ValidatedUpload`, содержащий:

- нормализованное имя файла;
- расширение;
- `contentType`;
- размер.

### `FilenameSanitizer`

Вспомогательный компонент для безопасной обработки имени файла.

По логике проекта он нужен для:

- устранения потенциально опасных имён;
- нормализации имени перед генерацией `objectKey`;
- отделения расширения от базового имени;
- снижения рисков path traversal и некорректного хранения.

### `ObjectKeyFactory`

Фабрика внутреннего пути хранения файла в `MinIO`.

Формирует путь по шаблону:

```text
{category}/{year}/{month}/{day}/{fileId}/{baseName}-{suffix}.{ext}
```

Где:

- `category` преобразуется в lowercase и `kebab-case`;
- дата берётся из `Clock`;
- `suffix` строится по первым символам checksum либо по части `UUID`;
- расширение сохраняется нормализованным.

Это важное архитектурное решение, потому что:

- структура хранения становится предсказуемой;
- уменьшается риск конфликтов имён;
- в object key нет прямой зависимости от бизнес-сущности;
- ключ пригоден для безопасного массового хранения большого числа объектов.

### `ValidatedUpload`

Компактный value object с результатом успешной валидации.

## 5.6. Пакет `bucket` — связь бизнес-категорий и физического хранилища

### `FileCategory`

Ключевой enum внешнего бизнес-контракта.

Поддерживаемые категории:

- `ORDER_DOCUMENT`
- `ORDER_ESTIMATE`
- `ORDER_INSPECTION_PHOTO`
- `VEHICLE_PHOTO`
- `VEHICLE_DOCUMENT`
- `CUSTOMER_DOCUMENT`
- `CUSTOMER_AVATAR`
- `EMPLOYEE_AVATAR`
- `INVOICE`
- `REPORT`

Каждая категория содержит:

- `bucketName`
- `group`

Группы:

- `AVATAR`
- `PHOTO`
- `DOCUMENT`
- `REPORT`

Именно здесь инкапсулировано правило: **внешний API говорит категорией, а сервис сам знает, в какой bucket положить файл**.

### `BucketInitializer`

Компонент инициализации bucket'ов при старте приложения.

Назначение:

- пройти по категориям;
- определить все необходимые уникальные bucket'ы;
- проверить существование bucket;
- при необходимости создать его.

Это уменьшает ручные действия при локальном запуске и делает сервис более автономным.

## 5.7. Пакет `config` — конфигурация приложения

### `FileStorageProperties`

Типизированная конфигурация с префиксом `app.file-storage`.

Содержит три больших блока настроек:

1. `minio`
   - endpoint
   - access key
   - secret key
   - флаг автоинициализации bucket'ов

2. `presign`
   - default expiry seconds
   - max expiry seconds

3. `validation`
   - map правил по каждой `FileCategory`
   - максимальный размер
   - разрешённые MIME-типы
   - разрешённые расширения

Особенно важно для диплома, что правила валидации здесь централизованы и параметризованы, а не захардкожены в контроллере.

### `MinioConfig`

Конфигурационный класс, который создаёт `MinioClient`.

Также в нём есть fail-fast проверка обязательных параметров окружения. Это помогает обнаружить неверную конфигурацию ещё при старте приложения.

## 5.8. Пакет `common` — исключения и унифицированная обработка ошибок

Содержит проектный слой ошибок:

- `ApiErrorResponse`
- `ConflictException`
- `DeletedFileException`
- `FileNotAvailableException`
- `FileNotFoundException`
- `InvalidFileException`
- `StorageUnavailableException`
- `GlobalExceptionHandler`

### `GlobalExceptionHandler`

Назначение:

- централизованно преобразовывать ошибки в JSON-ответ;
- поддерживать единый формат ошибок;
- возвращать корректные HTTP-коды;
- отдельно обрабатывать ошибки binding/validation и слишком больших файлов.

Преимущество такого подхода:

- API сервиса предсказуемо для клиента;
- фронтенд и другие сервисы могут стабильно разбирать ошибки;
- инфраструктурные и доменные ошибки не смешиваются.

## 5.9. Пакет `access` — заготовка под авторизацию и контроль доступа

### `FileAccessPolicy`

Это интерфейс, задающий boundary для будущих правил доступа:

- `assertCanUpload(...)`
- `assertCanRead(...)`
- `assertCanDelete(...)`

### `FileUploadContext`

Контекст загрузки, передаваемый в policy.

Содержит:

- категорию;
- тип владельца;
- ownerId;
- uploadedBy.

### `AllowAllFileAccessPolicy`

Текущая временная реализация, которая разрешает все действия.

С архитектурной точки зрения это очень важная деталь:

- сервис уже спроектирован с учётом будущей security-интеграции;
- но на текущем этапе проект находится на MVP-уровне и ещё не подключил реальную проверку доступа.

Для диплома это можно описывать как **расширяемую точку интеграции с Auth/Core**, ещё не доведённую до production-политики.

## 7. Схема базы данных

## 7.1. Liquibase

Проект использует `Liquibase` для управления схемой БД.

Основной changelog:

## 9. Жизненный цикл файла внутри системы

Ниже приведён типовой сценарий загрузки.

### 9.1. Сценарий upload

1. Клиент отправляет multipart-запрос.
2. `FileController` принимает параметры и передаёт их в `FileService`.
3. `FileService` валидирует `ownerId`.
4. `FileService` вызывает `FileAccessPolicy`.
5. `FileUploadValidator` проверяет размер, MIME, расширение и имя.
6. Генерируется `UUID` файла.
7. Вычисляется `SHA-256` содержимого.
8. По `FileCategory` выбирается `bucketName`.
9. `ObjectKeyFactory` создаёт внутренний `objectKey`.
10. В БД создаётся запись `FileMetadata` в начальном состоянии.
11. Через `ObjectStorageService` файл отправляется в `MinIO`.
12. При успехе статус меняется на `AVAILABLE`.
13. При ошибке статус меняется на `UPLOAD_FAILED` и выполняется cleanup объекта.
14. Клиент получает `FileMetadataResponse`.

### 9.2. Сценарий download

1. По `fileId` проверяется наличие записи.
2. Проверяется, что файл доступен и не удалён.
3. Вызывается `ObjectStorageService.download(...)`.
4. Контроллер возвращает поток в HTTP response.

### 9.3. Сценарий presigned URL

1. Находится доступный `FileMetadata`.
2. `PresignService` нормализует requested TTL.
3. `ObjectStorageService` строит временную ссылку.
4. Клиент получает URL и время истечения.

### 9.4. Сценарий delete

1. Находится сущность файла.
2. Проверяется право удаления.
3. Если файл уже удалён — повторный вызов не ломает контракт.
4. Статус переводится в `DELETED`.
5. Устанавливается `deletedAt`.

## 10. Валидация и правила хранения

Проект реализует несколько уровней защиты.

### 10.1. Валидация входных параметров

- `ownerId` не должен быть пустым;
- `page` и `size` валидируются аннотациями `@Min` и `@Max`;
- `PresignedUrlRequest` проходит DTO-валидацию;
- multipart upload ограничен размером на уровне Spring.

### 10.2. Валидация файла

Проверяются:

- пустой файл;
- максимально допустимый размер;
- допустимый MIME-тип;
- допустимое расширение;
- безопасное имя файла.

### 10.3. Валидация presigned TTL

`PresignService` гарантирует, что:

- TTL не меньше 60 секунд;
- если TTL не указан, берётся значение по умолчанию;
- если TTL слишком большой, он ограничивается `maxExpirySeconds`.
